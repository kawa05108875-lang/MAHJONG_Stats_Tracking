# 麻雀成績管理アプリ 実装手順

## 1. 実装方針

まずは「仲間内グループで4人東南戦の半荘を記録し、半荘結果と個人成績を見られる」状態をMVPとして作る。

最初から全機能を細かく作り込むのではなく、以下の順番で進める。

1. Next.jsとFirebaseの土台を作る
2. 認証とグループ管理を作る
3. プレイヤー管理を作る
4. 半荘作成と局入力を作る
5. 半荘結果計算を作る
6. 個人成績とランキングを作る
7. セキュリティルールとデプロイを整える

## 2. フェーズ1: プロジェクト初期設定

### 作業内容

- Next.jsプロジェクトを作成する。
- TypeScriptを有効にする。
- ESLint、Prettierなど最低限の開発環境を整える。
- Firebaseプロジェクトを作成する。
- Firebase AuthenticationでGoogleログインを有効にする。
- Cloud Firestoreを有効にする。
- Vercelデプロイ用の環境変数を整理する。

### 作成する主な設定

- Firebase設定ファイル
- 環境変数ファイル
- ルーティング構成
- 共通レイアウト
- 認証状態を扱うProvider

### 完了条件

- Next.jsアプリがローカルで起動する。
- Firebaseに接続できる。
- Googleログインとログアウトができる。
- ログインユーザーのuid、displayName、emailを取得できる。

## 3. フェーズ2: データモデルと共通ロジック

### 作業内容

- Firestoreに保存するデータ構造をTypeScript型として定義する。
- 麻雀ルール、半荘、局、プレイヤー、成績の型を作る。
- 点数計算や順位計算など、画面から独立したロジックを関数として作る。

### 優先して作る型

- UserProfile
- Group
- GroupMember
- Player
- Match
- Hand
- PlayerStats
- MatchRule
- ScoreDelta

### 優先して作るロジック

- 現在持ち点の計算
- 点数増減の合計チェック
- 半荘順位の計算
- ウマ、オカ、合計ポイントの計算
- 和了率、放銃率、トップ率などの集計

### 完了条件

- データの形がコード上で明確になっている。
- 点数計算と成績計算をUIに依存せずテストできる。
- 要件定義書のデータ項目とコード上の型に大きなズレがない。

## 4. フェーズ3: 認証とグループ管理

### 作業内容

- 未ログイン時はログイン画面へ誘導する。
- ログイン後、ユーザー情報をFirestoreのusersに保存する。
- グループ一覧画面を作る。
- グループ作成画面を作る。
- グループ作成時に作成者をgroupMembersへ登録する。
- グループごとの既定ルールを保存する。

### 画面

- ログイン画面
- グループ一覧画面
- グループ作成画面
- グループホーム画面

### 完了条件

- Googleログインしたユーザーがグループを作成できる。
- 自分が参加しているグループ一覧を見られる。
- グループホームへ移動できる。
- グループに初期ルールが保存される。

## 5. フェーズ4: プレイヤー管理

### 作業内容

- グループ内のプレイヤー一覧を表示する。
- プレイヤーを追加できる。
- プレイヤー名を編集できる。
- Firebaseユーザーと紐づかない手入力プレイヤーも作れるようにする。
- playerStatsの初期データを作成する。

### 画面

- メンバー管理画面
- プレイヤー追加フォーム
- プレイヤー編集フォーム

### 完了条件

- グループ内にプレイヤーを4人以上登録できる。
- プレイヤー名を変更しても過去成績との紐づきが壊れない。
- 半荘作成時にプレイヤーを選択できる状態になる。

## 6. フェーズ5: 半荘作成

### 作業内容

- 半荘作成画面を作る。
- 参加プレイヤー4人を選択できるようにする。
- 起家を選択できるようにする。
- グループ既定ルールを表示し、必要なら半荘ごとに上書きできるようにする。
- 半荘作成時にmatchesへ保存する。

### 保存する主な情報

- 対局日
- 参加プレイヤー4人
- 起家
- 適用ルール
- 現在局
- 本場
- 供託
- 状態

### 完了条件

- 4人を選んで半荘を開始できる。
- 開始時点で各プレイヤーの持ち点が25000点として扱われる。
- 作成した半荘の局入力画面へ移動できる。

## 7. フェーズ6: 局入力

### 作業内容

- 局入力画面を作る。
- 現在局、本場、供託、各プレイヤーの現在点数を表示する。
- 和了、流局、罰符の入力フォームを作る。
- 局結果をhandsへ保存する。
- 保存後にmatchesの現在局、本場、供託を更新する。
- 入力済みの局結果を一覧表示し、修正できるようにする。

### 和了入力

- ツモ、ロンを選択する。
- 和了者を選択する。
- ロンの場合は放銃者を選択する。
- リーチ者を選択する。
- 各プレイヤーの点数増減を入力する。

### 流局入力

- 各プレイヤーの聴牌、ノーテンを入力する。
- 各プレイヤーの点数増減を入力する。
- リーチ棒の供託を反映する。

### 罰符入力

- 対象プレイヤーを選択する。
- 理由メモを入力する。
- 各プレイヤーの点数増減を入力する。

### 完了条件

- 和了、流局、罰符の局結果を保存できる。
- 点数増減の合計が0でない場合に警告できる。
- 保存した局結果から現在点数を再計算できる。
- トビ終了条件を満たした場合、半荘終了へ進める。

## 8. フェーズ7: 半荘結果

### 作業内容

- 半荘結果画面を作る。
- 全局の点数増減から最終持ち点を計算する。
- 起家優先で同点順位を決める。
- ウマ10-30、25000点持ち30000点返しでポイントを計算する。
- 半荘を終了確定できるようにする。
- 終了時にmatchesのstatusとfinalResultsを更新する。

### 完了条件

- 最終持ち点、順位、素点、ウマ、オカ、合計ポイントが表示される。
- 半荘を終了確定できる。
- 終了した半荘がランキング、個人成績の集計対象になる。

## 9. フェーズ8: 個人成績とランキング

### 作業内容

- 局結果保存時、修正時、半荘終了時にplayerStatsを更新する。
- グループ内ランキング画面を作る。
- 個人成績詳細画面を作る。
- 合計ポイント、平均順位、和了率、放銃率などで並び替えできるようにする。

### 表示する主な指標

- 半荘数
- 参加局数
- 合計ポイント
- 平均ポイント
- 平均順位
- トップ率
- 連対率
- ラス率
- 和了率
- 放銃率
- 立直和了率
- ツモ率

### 完了条件

- グループ内のランキングを表示できる。
- プレイヤーごとの詳細成績を表示できる。
- 「放ラ率」は画面上では「和了率」として扱われる。
- 局や半荘を修正した場合、成績も更新される。

## 10. フェーズ9: Firestore Security Rules

### 作業内容

- ログインしていないユーザーのアクセスを拒否する。
- グループメンバーだけが対象グループのデータを閲覧できるようにする。
- グループメンバーだけが対象グループのデータを作成、更新できるようにする。
- usersは本人のデータを作成、更新できるようにする。

### 完了条件

- グループ外ユーザーがグループ内データを読めない。
- グループ外ユーザーが半荘、局、プレイヤーを作成、更新できない。
- Firebase Emulatorまたは手動確認で主要ルールを検証できる。

## 11. フェーズ10: UI調整とデプロイ

### 作業内容

- スマートフォン優先で画面を調整する。
- 対局中に必要な情報を上部に固定する。
- 入力ボタンや選択UIを押しやすくする。
- Vercelにデプロイする。
- Firebaseの本番環境変数をVercelへ設定する。

### 完了条件

- スマートフォンで局入力がしやすい。
- Vercel上でログイン、グループ作成、半荘入力、成績閲覧ができる。
- 本番FirestoreのSecurity Rulesが有効になっている。

## 13. 最初に作るべきMVP

最初のMVPでは、以下だけが動けばよい。

- Googleログインできる。
- グループを作れる。
- プレイヤーを4人登録できる。
- 半荘を作れる。
- 局ごとに点数増減を手入力できる。
- 現在点数を表示できる。
- 半荘終了時に順位とポイントを表示できる。
- プレイヤー別の基本成績を表示できる。

自動点数計算、詳細な役入力、グラフ、CSV出力は後回しにする。
